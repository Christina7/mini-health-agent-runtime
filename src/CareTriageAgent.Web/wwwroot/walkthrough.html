Architecture

A reusable core, a health domain, two thin hosts

The runtime knows nothing about health — it could host any agent. All health specifics live one layer up. Two hosts (a web app and a CLI) drive the exact same composition root; neither contains agent logic. That separation is the whole point: it's a runtime, not a triage script.

CareTriageAgent.Web  ·  CareTriageAgent.Cli
Thin hosts. The web host marshals HTTP ↔ runtime (POST /triage) and returns the serialized trace; the CLI feeds one message.
▼ both call the same CareTriageSession.OnUserMessageAsync
CareTriageAgent · the health domain
Symptom knowledge base, red-flag guardrail, deterministic planner, triage policy, shared data. The composition root wires the runtime to the domain.
▼ built on
AgentRuntime · domain-agnostic core
Orchestrator loop · tools · guardrail hook · config + JSON-Patch flights · failure framework · work context · OpenTelemetry tracing. Zero health knowledge.

How it was built

Test-first, in vertical slices

Each slice left the repo green and runnable; each was one merged PR. 41 unit + integration tests pin every behavior (xUnit + Moq + WebApplicationFactory), run on CI.

Run it

Clone & dotnet run — no keys, no network, no database

The only prerequisite is the .NET 8 SDK. The default path is fully offline and deterministic.

# build & test
$ dotnet build MiniHealthAgentRuntime.sln
$ dotnet test # 41 passing

# the web app — browser chat UI with a live trace tree
$ dotnet run --project src/CareTriageAgent.Web # open the printed http://localhost:5xxx

# or the CLI — same runtime, deterministic
$ dotnet run --project src/CareTriageAgent.Cli -- "severe chest pain and shortness of breath"

Architecture

A reusable core, two health domains, two thin hosts

The runtime knows nothing about health — it could host any agent. All health specifics live one layer up, in two sibling domains that never reference each other. Two hosts drive the same composition roots; neither contains agent logic. That separation is the whole point: it's a runtime, not a triage script — and the second agent is the proof.

HealthAgents.Web  ·  CareTriageAgent.Cli
Thin hosts. The web host marshals HTTP ↔ runtime and serves both agents (POST /triage + POST /plan); the CLI feeds one triage message. Neither contains agent logic.
▼ Web → both sessions  ·  CLI → CareTriageSession only
CareTriageAgent · single-step triage
Symptom KB, red-flag guardrail, deterministic planner, triage policy. One turn, root span triage.turn.
HealthPlanAgent · multi-step planner
profile_analyzer → plan_generator → task_decomposer (plus a log chain); typed PlanEnvelope, UnsafeGoalGuardrail. Root span plan.turn.
▼ both built on
AgentRuntime · domain-agnostic core
Orchestrator loop · tools · guardrail hook · config + JSON-Patch flights · failure framework · work context · OpenTelemetry tracing. Zero health knowledge.

How it was built

Test-first, in vertical slices

Each slice left the repo green and runnable; each was one merged PR. 74 unit + integration tests pin every behavior (xUnit + Moq + WebApplicationFactory), run on CI.

Run it

Clone & dotnet run — no keys, no network, no database

The only prerequisite is the .NET 8 SDK. The default path is fully offline and deterministic.

# build & test
$ dotnet build MiniHealthAgentRuntime.sln
$ dotnet test # 74 passing

# the web app — walkthrough at /, live chat app at /app
$ dotnet run --project src/HealthAgents.Web # open the printed http://localhost:5xxx
  # press Ctrl+C in this terminal to stop the server

# or the CLI — same runtime, deterministic
$ dotnet run --project src/CareTriageAgent.Cli -- "severe chest pain and shortness of breath"
